# PayPal buy fixed — it was a `ctx` scope bug, not creds

## What looked wrong vs what was wrong
`arcana proxy buy 5` → `Error: paypal_error`. Looked like bad/sandbox PayPal creds. It was
NOT. The real error (once surfaced) was **`ctx is not defined`**.

## Root cause
`getUser(auth, kv, ctx, env)` requires `ctx` (ExecutionContext, for `ctx.waitUntil` on the
KV/Cache writes). `handleCreateOrder` and `handleCaptureOrder` call `getUser(..., ctx, env)`
but were defined as `(request, env, cors)` — no `ctx` param → ReferenceError → the `catch`
collapsed it into a generic `{error:"paypal_error"}`. So buy AND capture failed despite
valid LIVE PayPal creds.

## Fix (deployed; commit 0d9226c)
- Pass `ctx` from the fetch dispatch into `handleCreateOrder` / `handleCaptureOrder`.
- Surface the real error: `getPayPalToken` throws PayPal status/error; create-order returns
  the underlying message + `console.error`s it (no creds leaked).
- Verified: `/v1/pay/create-order` → 200 + a live `approvalUrl`.

## Recurring pattern (3rd time)
Same class as license-server `seedKeys` (out-of-scope var) and the TUI worker env gotcha:
**a missing/unthreaded param + a broad `catch` = a real bug hidden behind a generic error.**
Lesson: don't let handlers swallow exceptions into opaque error codes — include the real
message (status + body), at least via `console.error` for `wrangler tail`.

## State
